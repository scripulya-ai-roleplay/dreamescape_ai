# Agent anti-hang: heartbeat + watchdog

How the backend stops waiting forever when `scripulya_agent` dies, stalls, or drops
a result — and how it tells a *legitimately slow* generation from a *dead* one.

This doc is written bottom-up: concrete failure → the trick that detects it → the
code that implements it → the principle behind it.

---

## 1. The problem (concrete)

Normal flow is fire-and-forget over RabbitMQ:

```
user sends message
   └─► backend publishes LLMRequest to the agent ──► returns HTTP 202 immediately
                                                     (does NOT wait for the reply)

   ...the model reply comes back later as an LLMResult on a second queue,
    and a backend consumer turns it into a chat message + pushes it over SSE.
```

That works — **until the agent dies mid-generation.** Now no `LLMResult` ever arrives:

```
user sends message  →  backend hands it to the agent  →  AGENT DIES
                                                                │
   backend is waiting for a reply that will never come  ◄───────┘
   the client SSE spinner spins FOREVER
```

The backend has no built-in way to notice the agent died. We add one.

---

## 2. The idea: a dead-man's switch

The agent must keep "pressing a button" every few seconds while it works. If it
stops, something declares the generation failed and tells the user.

That needs **two** separate jobs — which is exactly the shape of the code:

| Job | Done by | Class |
|---|---|---|
| press the button + record who's pressing | the agent + a Redis store | `RedisGenerationHeartbeat` (detect) |
| notice a button went silent + tell the user | a background loop | `GenerationWatchdog` (act) |

---

## 3. The cast

```
   scripulya_ai (backend)                  Redis                    scripulya_agent
   ────────────────────                  ────────                  ───────────────
   ScripulyaAgentClient.publish                                    Heartbeat (refreshes :alive
     └─ registers the generation ──►   chalkboard   ◄── on every tick while generating)
                                        (keys)       ──► mark_done on finish
   GenerationWatchdog (every 10s)
     └─ reads who's dead ──►
     └─ writes FAILED + SSE
```

Redis is the shared chalkboard both processes scribble on. Neither backend nor agent
needs to talk to the other directly to coordinate liveness.

---

## 4. The shared chalkboard: what's literally in Redis

For one in-flight generation with `request_id = "req-a"`:

```
gen:inflight          →  SET  { "req-a" }        type=set   the work list (who's outstanding)
gen:req-a:chat        →  "chat-123"             type=str   maps request → which chat to apologize to
gen:req-a:alive       →  "1"   (TTL 30s)        type=str   THE BUTTON. present = "still working"
gen:req-a:done        →  (absent)               type=str   "I'm finished" marker
```

- The **agent** writes: refreshes `:alive` every 10s; on finish sets `:done` and deletes `:alive`.
- The **backend** writes: on publish, adds to `inflight`, sets `:chat`, sets the first `:alive`.

`request_id` is a per-call UUID generated by the backend at publish time and carried
as the **RabbitMQ `correlation_id`**. The agent reads that same id back and uses it as
the key suffix — so liveness is correlated to the *right* generation, not just the chat.

---

## 5. The TTL trick: absence *is* the death signal

We **never** run code to delete `:alive` when the agent dies. We just:

1. `SET gen:{rid}:alive 1 EX 30` — the key will self-delete in 30s.
2. The agent re-runs that same `SET` every 10s → each refresh pushes the self-delete
   30s into the future.
3. If the agent dies → it stops refreshing → 30s later **Redis removes the key itself.**

So **"the key is gone" IS the death signal.** We detect death by *asking whether the
key still exists* — not by tracking time ourselves. No timers, no cleanup jobs, no
"last seen" comparisons. Redis's TTL does the bookkeeping.

---

## 6. The decision rule (the whole logic, distilled)

When the watchdog wakes up and looks at one `req-a`, there are exactly three possible
states and one action each:

```
  :alive present?    :done present?     →   action
  ──────────────────────────────────────────────────────────
       YES                *             →   SKIP     (still working — even if slow)
       NO                YES            →   DRAIN    (finished; just remove from work list)
       NO                 NO            →   DEAD  →  write a FAILED reply to that chat
```

That table *is* `sweep_dead()`. Everything else is bookkeeping.

---

## 7. Timelines (proving the rule)

**Happy path** — agent finishes in 5s:

```
t=0    publish    → SET :alive EX30
t=2    agent refreshing :alive
t=5    done       → SET :done ; DEL :alive        (reply delivered normally via RabbitMQ)
t=10   sweep:  :alive? no.  :done? YES → DRAIN, no FAIL   ✔
```

**Slow but alive** — a 3-minute max-effort generation (the case we must NOT break):

```
t=0 … 180s   agent refreshing :alive every 10s the whole time
every sweep: :alive? YES → SKIP   ✔   (never FAILed — it's slow, not dead)
t=180   done → :done set
```

**Dead agent** — the whole point:

```
t=0    publish → SET :alive EX30
t=8    AGENT CRASHES — stops refreshing
t=38   :alive's 30s TTL elapses → key vanishes (by itself)
t≤40   sweep:  :alive? no.  :done? no → DEAD
       → claim it (NX-SET :done="watchdog", so no other sweep double-handles it)
       → read :chat = "chat-123"
       → write a FAILED message to chat-123, push it over SSE
```

So a dead agent surfaces a failure within ≈ `alive_ttl (30s) + sweep (≤10s)` ≈ **≤40s**,
instead of forever. (Verified live: a killed agent produces a `FAILED` row within one
sweep of the heartbeat expiring.)

---

## 8. The code: detection vs action

Two classes, joined by a tiny contract:

```
   ┌─────────────────────────────────────────────────────┐
   │  GenerationWatchdog            — the ACTOR / driver │
   │  (src/application/llm_watchdog.py)                  │
   │  ───────────────────────────────────────────────    │
   │  run_forever():  loop  →  sweep_once()  →  sleep    │
   │  sweep_once():                                      │
   │     dead = heartbeat.sweep_dead()   ◄── "who died?"│
   │     for (rid, chat_id) in dead:                    │
   │        DB  ← write FAILED message(chat_id)          │
   │        SSE ← push it to the client                  │
   └───────────────────┬─────────────────────────────────┘
                       │ the only coupling:
                       ▼    list[(request_id, chat_id)]
   ┌─────────────────────────────────────────────────────┐
   │  RedisGenerationHeartbeat   — the DETECTOR / store  │
   │  (src/infrastructure/gateways/redis_heartbeat.py)   │
   │  ───────────────────────────────────────────────    │
   │  register_inflight(rid, chat)  ← backend, on publish│
   │  sweep_dead() -> [(rid, chat)] ← implements §6      │
   │  (knows ONLY Redis; knows nothing about DB or SSE)  │
   └─────────────────────────────────────────────────────┘
```

- **`RedisGenerationHeartbeat`** answers one question — *"which in-flight requests are
  dead?"* — with pure Redis queries. It can't *do* anything about a death; it reports it.
- **`GenerationWatchdog`** consumes that answer and *acts*: for each dead `(rid,
  chat_id)` it writes a `FAILED` row through `IMessageService` and fans it out over the
  SSE gateway. It knows nothing about Redis.

`RedisGenerationHeartbeat` is the implementation of the `IGenerationHeartbeat` port
(`src/application/ports.py`), so the storage is swappable.

---

## 9. The Redis commands used

| redis-py call | Redis command | Meaning | Returns |
|---|---|---|---|
| `set(k,v,ex=n)` | `SET k v EX n` | store a string that auto-deletes after `n`s | OK |
| `set(k,v,ex=n,nx=True)` | `SET k v EX n NX` | same, **only if `k` is absent** | `True`/`False` |
| `exists(k)` | `EXISTS k` | is the key there? | 1 / 0 |
| `get(k)` | `GET k` | read the string | value, or `None` |
| `sadd(s,m)` | `SADD s m` | add member to set | count added |
| `smembers(s)` | `SMEMBERS s` | all members of set | set of strings |
| `srem(s,m)` | `SREM s m` | remove member from set | count removed |
| `pipeline(transaction=True)` | `MULTI … EXEC` | run several commands as one atomic batch | all results |

**`register_inflight`** (one atomic batch):

```
MULTI
SADD gen:inflight req-a                       # add to work list
SET  gen:req-a:chat "chat-123" EX 1800        # remember the chat (hard-deadline backstop)
SET  gen:req-a:alive "1"      EX 45           # the button, grace window until the agent picks up
EXEC
```

**`sweep_dead`** (per key in `gen:inflight`):

```
EXISTS gen:req-a:alive           → 0                # expired ⇒ NOT alive
SET gen:req-a:done "watchdog" EX 3600 NX
                                 → OK  (claimed)    # …or (nil) = already done ⇒ skip
SREM gen:inflight req-a          → 1                # off the work list
GET  gen:req-a:chat              → "chat-123"       # who to tell
# return ("req-a", "chat-123") as DEAD
```

`SET … NX` ("set if Not eXists") is Redis's built-in **lock/claim**: if two sweeps (or a
sweep racing the agent's own `mark_done`) hit the same key at once, only the first gets
`OK`; the rest get `(nil)` and skip. A dead request can therefore be claimed only once —
no double-FAILED rows. No separate lock key; `:done` *is* the lock.

## 10. Why the watchdog loop doesn't freeze the backend

```python
async def run_forever(self) -> None:
    while True:
        try:
            await self.sweep_once()
        except Exception:
            self.logger.exception("watchdog sweep crashed")
        await asyncio.sleep(settings.LLM_SWEEP_INTERVAL_SECONDS)
```

There's a single event-loop thread. It freezes **only** on a *synchronous blocking* call.
`await asyncio.sleep(n)` is the opposite of blocking — it schedules a wakeup `n` seconds
later and hands the thread back immediately:

```
time.sleep(10)          # ❌ hogs the one thread for 10s — every request stalls
await asyncio.sleep(10) # ✅ "wake me in 10s"; thread is free to serve HTTP meanwhile
```

The watchdog is just one more task on the loop; it borrows the thread only at each
`await`, then yields. `sweep_once()` uses async Redis, async DB (asyncpg), and the SSE
publish is an in-memory `put_nowait` — so it never blocks either. Verified live: HTTP,
RabbitMQ, and SSE all served normally while sweeps ran every 10s.

The one rule: **never put a blocking call (`time.sleep`, `requests`, sync DB driver,
big CPU crunch) inside `sweep_once`** — that *would* freeze the backend.

---

## 11. The other half: provider-side timeout (livelock)

The heartbeat detects a **dead** agent (stops refreshing). It cannot detect a
**livelock**: the agent is alive and refreshing `:alive`, but the provider call itself
hangs and never returns. For that, the agent wraps the provider call in a hard timeout:

```python
# scripulya_agent/src/controllers/llm.py
async with Heartbeat(redis_client, correlation_id):
    async with asyncio.timeout(settings.LLM_GENERATION_TIMEOUT_SECONDS):   # 300s
        reply = await svc.handle(msg)
```

```
dead agent      →  heartbeat expires  →  watchdog FAILs        (within ~40s)
hung provider   →  5-min timeout       →  agent returns a       (within 5min)
                    (livelock)             generation_timeout
                                           error result
```

Together they cover both failure shapes; a legitimate slow generation (e.g. 3 min) is
untouched by either, because the heartbeat keeps refreshing and it finishes before 5 min.

---

## 12. How the failure actually reaches the user

Both the watchdog and the normal error path write a `FAILED` model message and push it
over SSE. For the client to *see* it, the SSE frame must use the standard `message`
event name — **not** `error` (which collides with `EventSource`'s reserved
connection-error event and would be silently dropped). See `_sse_frame` in
`src/application/events/server_events_service.py` and the `event: message` rationale
documented there.

## 13. Configuration

All knobs are env-overridable settings.

**Backend** (`scripulya_ai/src/conf.py`):

```
REDIS_URL                          = redis://redis:6379/0
LLM_HEARTBEAT_ALIVE_TTL            = 30     # no refresh for this long ⇒ agent considered dead
LLM_HEARTBEAT_GRACE_TTL            = 45     # initial alive TTL on submit; must be > ALIVE_TTL
LLM_HEARTBEAT_HARD_DEADLINE_SECONDS= 1800   # backstop TTL so inflight keys can't leak
LLM_SWEEP_INTERVAL_SECONDS         = 10     # how often the watchdog sweeps
```

**Agent** (`scripulya_agent/src/conf.py`):

```
REDIS_URL                              = redis://redis:6379/0
LLM_HEARTBEAT_ALIVE_TTL                = 30     # MUST match the backend, else the watchdog mistimes
LLM_HEARTBEAT_REFRESH_INTERVAL_SECONDS = 10
LLM_GENERATION_TIMEOUT_SECONDS         = 300    # provider-call hard cap (livelock)
```

> **Contract across repos:** the key names (`gen:inflight`, `gen:{rid}:{chat,alive,done}`)
> and `LLM_HEARTBEAT_ALIVE_TTL` are shared between the two apps. There is no shared
> package — keep them in sync manually.

Detection latency for a dead agent ≈ `alive_ttl + sweep_interval`.

## 14. Edge cases & limitations

| Case | Behavior |
|---|---|
| **Redis down** | All heartbeat ops are best-effort (swallowed + logged). Generation still works; only the safety net goes inactive. |
| **Backend restart** | State lives in Redis, not memory — the watchdog resumes and still catches a dead agent. (This is why state isn't an in-process dict.) |
| **Slow-recovering agent** | If a call exceeds `alive_ttl` (stalled), gets FAILed, then delivers late, a second `COMPLETED` may append after the `FAILED`. Rare; tune `alive_ttl`. |
| **Livelock** (agent alive, provider hung) | Not caught by the heartbeat; caught by the 5-min provider timeout. |
| **Result lost in transit after `:done` set** | Watchdog treats it as resolved; the reply may be missing. (RabbitMQ reliability — out of scope.) |
| **Mock mode** (`LLM_AGENT_ENABLED=false`) | No broker, no watchdog started, mock client has no heartbeat — Redis is never contacted. |

## 16. The principle

The whole design is **detection separated from action**, joined by one tiny contract:

```
sweep_dead() -> list[(request_id, chat_id)]
```

- The *detector* (`RedisGenerationHeartbeat`) is replaceable — swap Redis for anything
  that can answer "who's dead?"
- The *actor* (`GenerationWatchdog`) is replaceable — swap "write FAILED + SSE" for any
  other response.
- Neither references the other's internals — only that list.

And the detection itself rests on one elegant property: **a key with a TTL that is
continually refreshed is alive; the moment the refreshes stop, the key vanishes on its
own — and its absence is the signal.**
